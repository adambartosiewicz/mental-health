# Strengths

## Purpose

What actually works for me — durable capacities, conditions, and approaches that have demonstrably produced good outcomes. This file exists to **counterweight the negativity bias** that systems like this one can drift into, but only with evidence. A strength written here without supporting examples is a flattery, not an asset.

## Update rules

- A strength belongs here only when there are at least 2–3 concrete instances of it producing a real outcome.
- Distinguish:
  - **Capability** — something I can do well.
  - **Condition** — a context under which I function well (time of day, environment, type of work, type of relationship).
  - **Strategy** — a deliberate approach that has repeatedly worked.
- For each entry, name the **conditions under which the strength applies**. Few strengths are universal — most are situational.
- If a strength has gone unused for a long time, that is itself a data point worth noting.

## Entry template

```markdown
### <Strength name>

- **Type:** capability / condition / strategy
- **Description:** <plain language>
- **When it shows up:** <specific contexts>
- **When it doesn't:** <where it breaks down — equally important>
- **Evidence:** <session files, completed experiments, concrete outcomes>
- **Last reviewed:** YYYY-MM-DD
```

## Example

```markdown
### Long uninterrupted mornings

- **Type:** condition
- **Description:** Best work happens in 2–3 hour uninterrupted blocks before noon. Output quality drops sharply with even one interruption.
- **When it shows up:** Days starting before 8am with no meetings scheduled before noon.
- **When it doesn't:** Days with any morning meeting, even brief ones, derail the block entirely.
- **Evidence:** `sessions/2026-03-20-deep-work.md`, `experiments/2026-04-01-morning-block.md`
- **Last reviewed:** 2026-06-09
```
